1 · Prerequisite knowledge

Ten ideas that unlock the whole project

On-device LLM

ChatGPT runs in a data center. A small language model (1–3 billion parameters instead of trillions) can run on the phone itself — no internet, no account, fully private. Slower and less capable, but yours.

Quantization & GGUF

A model is billions of numbers. At full precision each takes 4 bytes → a 1B model would be ~4 GB. Quantization compresses each number to ~4 bits, shrinking the model to ~700 MB with minor quality loss. Q4_0, Q4_K_M, Q8_0 are different compression schemes; GGUF is the file format that stores them.

Prefill vs decode

Two phases of answering. Prefill (pp): the model reads your prompt — parallel work, can be very fast. Decode (tg): it writes the answer one token at a time — sequential, always slower. We measure both in tokens per second; a token ≈ ¾ of a word.

Matrix multiplication

Nearly all LLM compute is multiplying huge grids of numbers together, billions of times per answer. Whoever does matmul faster, wins. Everything below is about making matmul faster.

SIMD instructions: dotprod & i8mm

Arm chips include special instructions that crunch many numbers in one step. dotprod (common) is a good shortcut; i8mm (newer, Armv8.6+) is a much better one for 8-bit integer matmul — exactly what quantized models need. Our Nothing 2a has i8mm; a Pixel 7a doesn't.

The -march compiler flag

Compilers default to safe, generic code that runs on any arm64 chip — and uses none of the fast instructions. -march=armv8.2a+i8mm+dotprod is permission to emit them. This single flag was worth 4.94× on prefill in our measurements. Omit it and the silicon idles.

big.LITTLE cores

Phone CPUs mix fast, power-hungry cores with slow, efficient ones. The 2a: 2× fast A715 + 6× slow A510. Counter-intuition: using all 8 cores makes generation slower than using just the 2 fast ones — the slow cores become the bottleneck everyone waits for.

KleidiAI

Arm's official library of hand-tuned matmul routines, meant to exploit dotprod/i8mm/SME2. The obvious assumption — "turn it on, get speed" — is what we tested. Result on this chip, for Q4_0: no measurable gain once the compiler flags are right. Plain compiled i8mm code already saturates the hardware.

llama.cpp & llama-bench

llama.cpp is the open-source C++ engine that actually runs GGUF models on CPUs — the industry standard for local inference. llama-bench is its built-in stopwatch: it loads a model, runs fixed workloads, and reports tok/s as JSON.

adb & cross-compilation

adb (Android Debug Bridge) is a USB command line to the phone: push files, run programs, read sensors. Cross-compilation is building arm64 phone binaries on an x86 laptop — the Android NDK provides the compiler that makes this possible. No Mac required.

2 · Architecture

From TypeScript down to the silicon

The product is a stack: each layer only talks to the one below it. The user never sees anything below the first row — but every claim on the landing page was produced by the bottom three.

App · TypeScript
React Native app — chat UI, the "optimization lab" (sweep + results screens), model downloads. All authored code is TypeScript — by design, no C++ or Kotlin is ever written in this project.
Bindings
llama.rn — a prebuilt bridge that lets JavaScript call llama.cpp. The open question we're investigating: do its prebuilt binaries include the arch flags? If not, every RN-based LLM app ships the slow path.
Engine · C++
llama.cpp — loads the GGUF model, runs the matmuls. We compile it ourselves with the NDK, once per configuration under test (7 variants so far).
Silicon
Arm CPU — Dimensity 7200 Pro — 2× Cortex-A715 @ 2.8 GHz + 6× Cortex-A510 @ 2.0 GHz, with dotprod + i8mm + SVE2. The fixed reality every layer above must adapt to.

Alongside the stack, three support components that never ship in the app but produce all the evidence:

armcreate/
├── app/ ← the React Native app (Week 2+)
├── harness/ ← bench.py — pushes builds+model to the phone, times them, saves JSON
├── models/ ← the .gguf compressed models (Q4_0 · Q4_K_M · Q8_0), ~0.7–1.3 GB each
├── results/ ← raw measurement JSON — the evidence behind every published number
├── vendor/llama.cpp/ ← the engine source + 7 compiled build variants (build-android-*)
├── site/ ← this website
├── docs/ ← project brief
├── PLAN.md ← the full project plan
└── CLAUDE.md ← context so AI-assisted sessions resume with full state

4 · Method

Seven builds, one variable at a time

You can't attribute a speedup unless exactly one thing changes between measurements. So llama.cpp is compiled seven ways, and consecutive comparisons isolate each lever:

Build variant-march flagsKleidiAIRepackQuestion it answers
genericnoneoffonWhat do you get if you don't think about any of this? (The default.)
arch+i8mm+dotprodoffonWhat are the compiler flags alone worth?
kleidiai+i8mm+dotprodononDoes Arm's library add anything on top?
arch-norepack+i8mm+dotprodoffoffBaseline with llama.cpp's own weight-repacking disabled.
kleidiai-norepack+i8mm+dotprodonoffKleidiAI's true standalone value, unmasked by repack.
dp-arch+dotprod onlyoffonTarget for phones without i8mm (built, unused since going 2a-only).
dp-kleidiai+dotprod onlyononSame, with KleidiAI.

Measurement rules (non-negotiable): physical phone only, never an emulator · 5 repetitions per config · fixed workloads (128-token prefill, 64-token decode) · 90–120 s cooldowns between variants so heat from one run can't slow the next · battery level and temperature logged before and after every variant · raw JSON committed to results/ so every published number is auditable.

5 · Findings so far

What the phone actually said

BuildPrefill pp128Decode tg64Verdict
generic20.51 tok/s12.73 tok/sThe default everyone ships. Slow.
arch101.43 tok/s17.00 tok/s4.94× / 1.34× — the compiler flag is the whole gain.
kleidiai100.09 tok/s15.61 tok/sWithin noise of arch. No added value here.
arch-norepack103.90 tok/s17.33 tok/sEven repack adds nothing once i8mm is on.
kleidiai-norepack98.09 tok/s16.76 tok/sKleidiAI ≈ plain i8mm code, four ways tested.

Plus the thread-count result: decode at 2 threads (17.3 tok/s, pinned by the scheduler to the two fast A715 cores) beats 6 threads (13.8). More parallelism is worse when the extra workers are slow cores.

Honest caveats: single-session runs with ~5–8% thermal variance — error bars need more repetitions before these claims go in a submission. All numbers are one SoC (Dimensity 7200 Pro), one model (Llama 3.2 1B), one quant family so far. "KleidiAI adds nothing" means on this hardware, for this format — on SME2 silicon or other quants the answer may flip, which is exactly why the product measures instead of assuming.

Next experiments: the quantization sweep (Q4_0 vs Q4_K_M vs Q8_0 — downloaded, queued), energy per token via the battery sensor, big.LITTLE speculative decoding, and the llama.rn prebuilt-binary audit.
