Read it top to bottom: the entire speedup arrives with the compiler flags — permission for the compiler to use the chip's fast matrix instructions. The specialized library adds nothing after that. Most Android builds of llama.cpp never pass those flags.

CH2 · WHAT WE FOUND

Three results the folklore gets wrong

FINDING 01
4.94×

THE COMPILER FLAG IS THE WHOLE BALLGAME

One build flag — -march=armv8.2a+i8mm+dotprod — nearly quintuples prompt processing. It simply lets the compiler emit the matrix instructions the silicon already has. Most builds don't.

FINDING 02
+0%

KLEIDIAI ADDS NOTHING HERE

Arm's own acceleration library, tested four ways — with and without llama.cpp's repack path: no measurable gain on this SoC for Q4_0. Once i8mm is enabled, plain compiled code already saturates the hardware.

FINDING 03
2>6

FEWER THREADS, FASTER ANSWERS

Decode at 2 threads: 17.3 tok/s. At 6 threads: 13.8. The phone has 2 fast cores and 6 slow ones — spreading work across all of them drags the fast cores down to little-core pace.

Every one of these contradicts what a reasonable developer would assume. That's the thesis: your phone's fastest configuration is an empirical question — so PocketTune answers it empirically.
