Read it top to bottom. The entire speedup arrives with the compiler flags — permission to use the matrix instructions the silicon already has. The specialized library adds nothing after that. Most Android builds of llama.cpp never pass those flags.

What we found

Three results the
folklore gets wrong.

4.94×

The compiler flag is the whole ballgame.

One build flag nearly quintuples prompt processing. It simply lets the compiler emit the matrix instructions the chip already has. Most builds don't.

+0%

KleidiAI adds nothing here.

Arm's own acceleration library, tested four ways. No measurable gain on this chip for Q4_0 — once i8mm is on, plain compiled code already saturates the hardware.

2 > 6

Fewer threads, faster answers.

Decode at 2 threads: 17.3 tok/s. At 6 threads: 13.8. Two fast cores beat two fast plus six slow — the slow ones become the bottleneck everyone waits for.

Every one contradicts what a reasonable developer would assume. That's the point: your phone's fastest configuration is an empirical question — so PocketTune answers it empirically.
