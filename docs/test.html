What it does

Detect, sweep, apply, chat

Some Arm cores have i8mm matrix instructions, some only dotprod; some SoCs pair two big cores with six small ones. The "right" quantization, kernel path, and thread layout differ per device — and almost nobody tunes for it. PocketTune closes the loop:

01 · DETECT

Read the silicon

CPU features (dotprod, i8mm, SVE2) and core topology from /proc/cpuinfo. No native code.

02 · SWEEP

Benchmark on-device

Quant × KleidiAI kernels × threads. Measures prefill & decode tok/s, TTFT, RAM, and energy per token.

03 · APPLY

Install the winner

Downloads the optimal build and locks in the config — scored on perplexity as well as speed.

04 · USE

Offline assistant

A private chat app running the tuned setup. The optimization is the product.

Novelty headline

Big.LITTLE speculative decoding

The Dimensity 7200's six little cores usually idle during inference. PocketTune runs a tiny draft model on the little cores proposing tokens while the main model verifies on the big cores — llama.cpp's existing speculative mode plus thread affinity, pure configuration. No public per-phone data on this exists; ours will be first. (The Pixel 7a is a tri-cluster — A55 / A78 / X1 — so it offers a third affinity tier to test against.)

Battery awareness

Tokens per joule, not just per second

The track text names battery awareness; nobody publishes per-config energy data for on-device LLMs. PocketTune reads battery current via BatteryManager during every sweep and can recommend by answers-per-charge instead of raw speed.

Positioning

Why this isn't "another benchmark app"

PocketPal tells you a number. PocketTune makes your phone faster.

PocketPal AI (500K+ installs, same RN + llama.rn stack — proof our stack scales) benchmarks and ranks devices. PocketTune closes the loop: detect → sweep → recommend → apply, and adds the two datasets nobody publishes — per-config energy and big.LITTLE speculative decoding. No leaderboard here; that space is taken.
